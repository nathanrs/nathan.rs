### Kimi K1.5 [link] (Jan 2025)
Kimi k1.5 establishes a simplistic, effective RL framework without relying on more complex techniques such as Monte Carlo tree search, value functions, and process reward models, yet achieves state-of-the-art reasoning performance. The core thesis is that scaling context length is a substitute for explicit search machinery.
Policy optimization: online mirror descent
Rather than PPO or GRPO, they use a variant of online mirror descent. At each iteration i, they use the current model π_θi as a reference model and optimize a relative entropy regularized objective: maximize expected reward minus a KL penalty against the current reference. This has a closed-form optimal solution, which they use to derive a surrogate loss — effectively a squared error between the reward-adjusted log ratio and the empirical mean reward.
The gradient ends up looking like GRPO (mean reward as baseline, group of sampled responses), with two key differences:
Responses are sampled from π_θi rather than on-policy, and an L2-regularization term is applied to the log ratio rather than the asymmetric PPO clip. Since each iteration considers a different optimization problem due to the changing reference policy, they also reset the optimizer at the start of each iteration.
This is the GLM-style optimizer reset you asked about earlier — the motivation is explicit here: the optimization landscape changes when the reference model is swapped, so Adam's accumulated moment estimates from the previous iteration are stale and potentially harmful.
Long context scaling to 128k — the key architectural bet
They scale the context window of RL to 128k tokens and observe continued improvement with increased context length. The key insight is that an increased context length has the effect of increasing the number of search steps — so the model learns planning, reflection, and backtracking implicitly through its context, without needing an explicit search tree.
This is a conceptual reframe: instead of MCTS building a search tree externally, the long context is the search tree, flattened into a sequence. The model learns to write "wait, that approach was wrong" and backtrack within its own generated text.
Partial rollouts — solving the efficiency problem of long CoT
Long contexts are expensive to generate from scratch each RL iteration. Partial rollouts establish a fixed output token budget per iteration. If a trajectory exceeds the token limit, the unfinished portion is saved to a replay buffer and continued in the next iteration. Previous segments can be efficiently reused from the buffer — only the current iteration requires on-policy computation — significantly reducing overhead while allowing much longer responses.
They also add repeat detection: the system identifies repeated sequences in generated content and terminates them early, with detected repetitions assigned additional penalties to discourage redundant content generation.
Length penalty design
They observe an overthinking phenomenon where response length increases significantly during RL training. Their length reward promotes shorter correct responses and penalizes longer ones, while explicitly penalizing long incorrect answers. The length penalty is warmed up gradually — standard policy optimization runs first with no penalty, followed by a constant length penalty for the remainder of training.
The formula is relative within a group: a response's length reward is based on where its length falls between the min and max length of that group's samples, not an absolute token count.
Prompt curation — anti-reward-hacking
They exclude multiple-choice, true/false, and proof-based questions to prevent false positive verification. To remove easy-to-hack prompts, they prompt a model to guess answers without any chain-of-thought reasoning — if the model gets it right within N=8 attempts, the prompt is removed as too easy to game.
Difficulty is measured empirically: an SFT model generates answers 10 times per prompt at high temperature, and the pass rate becomes the difficulty proxy — lower pass rate means harder problem.
Long2Short — distilling reasoning efficiency
This is a notable secondary contribution: using the long-CoT model to improve short-CoT models. Four methods:
Model merging: simply averaging the weights of a long-CoT model and a short-CoT model, with no training. Shortest rejection sampling: sample the same question 8 times, keep the shortest correct response for SFT. DPO: use the shortest correct response as positive, longer responses (including correct ones that are 1.5× longer) as negatives. Long2short RL: a separate RL training phase with aggressive length penalties and reduced maximum rollout length.
The short-CoT version achieves 60.8 on AIME 2024 and 94.6 on MATH500, outperforming GPT-4o and Claude Sonnet 3.5 by up to 550%.
Reward modeling for math
They train two reward models: a classic value-head RM achieving ~84.4% accuracy on manual spot checks, and a chain-of-thought RM that generates step-by-step reasoning before outputting a correctness judgment — reaching 98.5% accuracy. They use the CoT RM in actual RL training.

The big picture
The paper's central argument is that long context + simple RL + good data curation can match systems that use expensive search infrastructure. The optimizer reset and mirror descent formulation keep training stable as the reference policy keeps shifting, partial rollouts make 128k-context RL computationally feasible, and the long2short pipeline means you get both a powerful reasoning model and an efficient deployment model from the same training run.

### Kimi K2 [link] (Jul 2025)
MuonClip optimizer
We've already gone deep on this, but to place it in context: using MuonClip, K2 was pre-trained on 15.5 trillion tokens with zero loss spikes throughout the entire training run. This is the headline engineering result — a 1 trillion parameter model trained to completion without a single training instability event, which at this scale is genuinely unusual.The two components are Muon (polar factor updates for ~2× token efficiency over AdamW) and QK-Clip (per-head rescaling of Q and K weights when attention logits exceed threshold τ=100, applied only to the heads that actually exhibit the problem).
Architecture: ultra-sparse MoE with empirically derived scaling laws
K2 uses 384 total experts with 8 activated per token (sparsity of 48), compared to 256 experts in DeepSeek-V3. Under the compute-optimal sparsity scaling law, achieving the same validation loss at sparsity 48 reduces FLOPs by 1.69× compared to sparsity 8.The key insight here is that they ran controlled scaling law experiments specifically for MoE sparsity under Muon — this is new because prior sparsity scaling laws were derived under AdamW, and the optimal sparsity may differ between optimizers. They found that higher sparsity consistently wins under fixed activated FLOPs, up to infrastructure complexity limits.They also cut attention heads from 128 to 64. At 128k sequence length, doubling attention heads causes an 83% increase in inference FLOPs, while controlled experiments show doubling heads only improves validation loss by 0.5–1.2%. Given sparsity 48 already provides strong performance, the marginal quality gain doesn't justify the inference cost — especially for an agentic model that needs efficient long-context processing.The architecture as a whole reflects a deliberate tradeoff: sacrifice a small amount of attention capacity to invest in more experts, optimizing for the agentic use case where long contexts and fast inference matter more than marginal attention expressivity.
Token utility via data rephrasing
Rather than repeating data (which causes overfitting) or simply collecting more, K2 introduces a synthetic rephrasing pipeline to amplify high-quality tokens without overfitting. Using style- and perspective-diverse prompting, chunk-wise autoregressive generation, and fidelity verification, they generate semantically equivalent rephrasings of knowledge-intensive documents. Testing on SimpleQA: raw wiki-text repeated 10 epochs scores 23.76, rephrasing once and repeating scores 27.39, rephrasing 10 times with a single pass scores 28.94. The chunk-wise autoregressive approach is notable — they split long documents into 256-token chunks, rephrase each with access to the previous chunks as context, then stitch them back together. This avoids the common problem of LLMs losing coherence in long rewrites.
Large-scale agentic data synthesis pipeline
This is the post-training equivalent of the data scaling contribution. The challenge: agentic tool-use behavior (multi-step planning, tool invocation, error recovery) is rare in natural data and expensive to collect from humans.They built a three-stage pipeline: first constructing a tool repository of 3000+ real MCP tools fetched from plus 20,000+ synthetically evolved tools; then generating thousands of distinct agent configurations and rubric-annotated tasks; then simulating multi-turn trajectories through a tool execution environment that maintains persistent state and introduces controlled stochasticity. Only trajectories passing rubric-based success criteria are retained. The tool simulator acts as a world model — it executes tool calls, tracks state across turns, and generates realistic failure cases (partial failures, edge cases, errors). For scenarios where simulation fidelity is critical, particularly in coding and software engineering tasks, they complement simulation with real execution sandboxes that run actual code against genuine development environments, providing ground-truth feedback through test suite pass rates. This hybrid approach — massive simulated diversity plus real-environment grounding for high-stakes tasks — is what enables the strong SWE-bench results.
Joint RL framework: RLVR + Self-Critique Rubric Reward
This is the most conceptually novel post-training contribution. The problem they're solving: RLVR (verifiable rewards from code execution, math checking etc.) works extremely well for objective tasks, but most of what makes an assistant useful — helpfulness, creativity, reasoning depth, tone — can't be verified with a rule. Their solution is a two-track RL system running jointly: Track 1 — Verifiable Rewards Gym: A Gym-like extensible framework with binary reward functions (1 for correct, 0 for incorrect) across math, STEM, logic, complex instruction following, faithfulness, coding, and safety. Standard RLVR. Track 2 — Self-Critique Rubric Reward: The model evaluates its own outputs to generate preference signals. The framework uses three types of rubrics: core rubrics representing fundamental values (clarity, conversational fluency, objective interestingness), prescriptive rubrics designed to prevent reward hacking (e.g., responses must not begin with compliments like "That's a great question"), and human-annotated rubrics for specific instructional contexts. The self-critique loop works as: K2 generates multiple responses to a subjective prompt, then K2 (acting as critic) scores pairs of responses against the rubrics, and the better response becomes a positive training signal. The key is that the critic model is continuously refined using verifiable signals — on-policy rollouts from verifiable-reward tasks are used to update the critic, distilling objective performance signals from RLVR into the evaluation model. This grounds subjective judgments in verifiable data, allowing gains from objective tasks to improve the critic's judgment on tasks without explicit reward signals. This is a closed loop: better performance on math/code → better critic → better judgments on writing/helpfulness → better policy on subjective tasks. The verifiable and non-verifiable tracks reinforce each other rather than being independent.
Infrastructure: flexible parallelism and activation management
Rather than optimizing a single parallelism strategy, they pursue a flexible configuration that allows K2 to be trained on any number of nodes that is a multiple of 32, using 16-way pipeline parallelism with virtual stages, 16-way expert parallelism, and ZeRO-1 data parallelism. This consistent design is important for research efficiency, as it substantially accelerates experimental iteration. For memory, selective recomputation is applied to LayerNorm, SwiGLU, and MLA up-projections — inexpensive to recompute but high memory footprint to store. Inputs of MoE up-projections and SwiGLU are compressed to FP8-E4M3 for storage. All remaining activations are offloaded to CPU RAM, with a copy engine streaming offload and onload overlapped with computation. The notable detail is that they avoid DualPipe (used by DeepSeek-V3 to overlap computation and communication) because at 1T+ parameters the memory overhead of DualPipe — which doubles parameter and gradient memory — becomes prohibitive. Instead they overlap EP communication with computation through careful scheduling of the interleaved 1F1B pipeline.

### Kimi K2.5 [link] (Feb 2026)
K2.5 is built on top of K2 (the 1T parameter MoE model we discussed) and extends it in two major directions: native multimodal capability (text + vision) and parallel agent orchestration. It's released as an open-weight model.
Joint optimization of text and vision: The central insight of K2.5's multimodal approach is that text and vision should be trained together throughout, not treated as separate concerns.
Early fusion beats late fusion. Conventional wisdom suggests introducing vision tokens predominantly in later stages at high ratios like 50% or higher. However, experiments reveal that early fusion with a lower vision ratio yields better results given a fixed total vision-text token budget. arxiv K2.5 mixes text and vision tokens at a constant moderate ratio throughout the entire 15T token pretraining process.
MoonViT-3D. The vision encoder processes images at native resolution using patch packing (no resizing distortion), and handles video by treating four consecutive frames as a spatiotemporal volume through shared weights. This means image understanding transfers directly to video without separate video-specific parameters.
Zero-vision SFT. Rather than using manually annotated visual SFT data (which is scarce and low diversity), zero-vision SFT uses only text SFT data to activate visual and agentic capabilities. All image manipulations are proxied through programmatic operations in IPython. This zero-vision activation enables diverse reasoning behaviors including pixel-level operations and generalizes to visually grounded tasks such as object localization, counting, and OCR. arxiv This works because joint pretraining already established strong vision-text alignment — the SFT just needs to install the behavioral pattern, which abundant text data can do.
Visual RL improves text performance. Outcome-based visual RL produced measurable improvements in textual tasks including MMLU-Pro (84.7% → 86.4%), GPQA-Diamond (84.3% → 86.4%), and LongBench v2 (56.7% → 58.9%). arxiv The bidirectional enhancement — text bootstraps vision, vision refines text — is the key finding of the multimodal training approach.
Policy optimization: K2.5's RL objective is a hybrid of K1.5's online mirror descent (L2 regularization on the log ratio) and DAPO's token-level normalization, with a modified clipping rule. The clipping mechanism functions as a gradient masking scheme: policy gradients are computed normally for tokens with log-ratios within the interval [α, β], while gradients for tokens falling outside this range are zeroed out. A key distinction from standard PPO clipping is that the method relies strictly on the log-ratio to bound off-policy drift, regardless of the sign of the advantages. Rewards are a mix of rule-based verifiable rewards for objective tasks, task-specific fine-grained visual rewards (IoU for grounding, edit distance for OCR, Gaussian-weighted distance for point localization), and Generative Reward Models (GRMs) extending K2's self-critique rubric approach to multimodal agentic trajectories.
Toggle — token-efficient RL: K2.5 introduces Toggle to reconcile reasoning capability with token efficiency. Toggle alternates between two phases every m iterations: Phase 0 (budget-limited) trains the model to solve problems within a task-dependent token budget, enforced only when mean accuracy exceeds threshold λ to prevent sacrificing quality for efficiency; Phase 1 (standard scaling) generates responses up to the maximum token limit, encouraging inference-time scaling. The problem-dependent budget is estimated from the ρ-th percentile of token lengths among correct responses. Toggle decreases output tokens by 25-30% with negligible performance impact, and generalizes across domains — training on math and code still reduces token usage on GPQA and MMLU-Pro.
Agent Swarm and PARL: The second major contribution is moving from sequential agent execution to parallel orchestration. The problem. Even models capable of hundreds of reasoning steps suffer from linear scaling of inference time. For complex tasks involving massive research, design, and development, sequential execution becomes impractical. Agent Swarm. K2.5 trains an orchestrator that dynamically decomposes tasks into subtasks and delegates them to specialized frozen subagents running in parallel. Subagents are frozen and their execution trajectories are excluded from the optimization objective — only the orchestrator is updated via RL. This decoupling circumvents credit assignment ambiguity and training instability. PARL reward. The reward has three components: task performance r_perf, a parallelism instantiation reward r_parallel (to prevent serial collapse where the model defaults to single-agent execution), and a subagent finish rate reward r_finish (to prevent spurious parallelism where the model spawns many useless subagents). The hyperparameters λ₁ and λ₂ on the auxiliary rewards are annealed to zero over training so the final policy optimizes purely for task performance. Critical steps metric. Rather than measuring total steps, K2.5 measures critical steps — the longest path through the parallel execution graph, analogous to the critical path in a computation graph. This explicitly incentivizes balanced task decomposition that minimizes end-to-end latency rather than just maximizing concurrency.Results. Agent Swarm reduces inference latency by up to 4.5× while improving item-level F1 from 72.8% to 79.0% compared to single-agent baselines.
Infrastructure: Decoupled Encoder Process: Training a multimodal model with pipeline parallelism creates load imbalance because the vision encoder sits at stage 0 of the pipeline and its compute varies with image count and resolution. K2.5 solves this with DEP: run the vision encoder forward pass first across all GPUs (load-balanced by patch count), discard intermediate activations, run the main backbone, then recompute the vision encoder backward pass. K2.5 achieves multimodal training efficiency of 90% relative to text-only training. 
The through-line: The paper's central thesis is that modality boundaries are artificial constraints. Text and vision should be trained together from the start, RL domains should be organized by capability rather than modality, and reward signals should flow across modalities. The practical result is a model where vision RL improves text benchmarks, text SFT activates visual capabilities, and a single unified RL environment handles both — rather than separate specialist pipelines that fight each other.

### Code World Model [link] (Sep 2025)
What CWM is and the core thesis
CWM (Code World Model) is a 32B dense decoder-only LLM from Meta FAIR that challenges a fundamental assumption in how LLMs learn to code. Standard pretraining treats code like any other text — predict the next token, left to right. To master coding, one must understand not just what code looks like but what it does when executed. Teaching LLMs code world modeling capabilities is typically not considered before post-training. The bet: if you train a model to simulate code execution — to predict what happens to variables when each line runs — it will develop a deeper understanding of code semantics rather than just syntax, and this will transfer to better code generation.
Python execution tracing at massive scale
The most novel data contribution is training on observation-action trajectories from the Python interpreter — structured records of exactly what happens when code executes, step by step.
Actions are Python statements and observations contain the contents of the local variables. By training CWM on a trajectory of observation-action pairs conditioned on the code only as context, the model is directly taught how the execution of a line of Python affects the state of local variables. The format is a JSON-like structure with custom tokens: each step records the current local variable state, then the line being executed, then the resulting variable state. Variables that don't change get an ellipsis to keep sequences compact.
Three sources of trace data:
Function-level tracing. Over 120M traced Python functions collected from online sources, with input-output pairs generated using fuzzing and Llama3-70B-Instruct. arxiv
CodeContests solutions. 33k competitive programming solutions traced against unit test inputs, including both correct and incorrect submissions — deliberately keeping failures to teach the model what incorrect execution looks like.
Repository-level tracing. Python execution tracing for unit tests of more than 21k repository images, with up to 40 historical commits traced per repository and capped at 4 commits to avoid overrepresentation. arxiv
Natural language tracing. A fourth variant where Qwen3-32B rewrites execution traces as natural language explanations rather than strict JSON format. This less-structured format allows for injection of semantic context (e.g., "this operation preserves the structure property of the max heap") or compressing traces by dynamically skipping less interesting parts. arxiv 75M trajectories from standalone functions plus 110k from CodeContests.
ForagerAgent: agentic mid-training data at scale
The second major data contribution is large-scale agentic trajectory data generated during mid-training rather than post-training — which is unusual.
ForagerAgent collects multi-step trajectories by prompting an LLM with a software engineering task to solve in the context of a particular code repository. Exposing CWM to such data at large scale early on should improve subsequent post-training in similar environments, as model predictions should already be grounded in environment dynamics. arxiv
Two task types:
Mutate-fix tasks. Start with a working codebase, synthetically introduce a bug (by removing functions, swapping variables, replacing operators, etc.), and have the agent fix it. Only mutations that cause unit tests to fail are kept — guaranteeing verifiable correctness.
Issue-fix tasks. Real GitHub issues from actual repositories, checking out commits before bug-fixing PRs and tasking the agent with resolving failing tests.
3M trajectories obtained from 10.2k images across 3.15k repositories, split 55-45 between issue-fix and mutate-fix tasks. Trajectories are deduplicated using MinHash with Jaccard similarity threshold 0.5 to avoid overfitting to repetitive interactions. arxiv
Notably, the model is trained on both successful and failed trajectories — the goal is learning the world model of the environment, not just imitating successful solutions.
Executable repository images infrastructure
Both the tracing and agentic data depend on being able to actually run arbitrary GitHub repositories at scale. Running both RepoAgent and Activ methods in parallel, over 35k unique executable repository images were created. arxiv
The Activ pipeline is clever — it repurposes GitHub Actions CI execution to build repository images locally, modifying the CI workflow to capture build state at test execution time. This is more reliable than documentation-based setup because machine-targeted CI instructions must remain accurate or builds fail immediately.
Architecture: sliding window attention for long context
CWM uses an alternating pattern of local and global attention blocks in a 3:1 ratio. Local sliding window attention covers 8,192 tokens, global attention covers 131,072 tokens, with GQA using 48 query heads and 8 key-value heads. arxiv
This is similar to the Mistral/Gemma approach — most attention is local (cheap), with occasional global attention layers for long-range information. The 3:1 ratio means 3 local layers per global layer, keeping the KV cache manageable while supporting 131k context for long agentic trajectories and repository-level code.
The training pipeline
Four stages: general pre-training on 8T tokens at 8k context, code world model mid-training on 5T tokens at 131k context, supervised fine-tuning on 100B tokens at 32k context, and joint reinforcement learning on 172B tokens at 131k context covering agentic and reasoning environments. arxiv
The mid-training stage is where CWM diverges from standard LLM pipelines — it introduces the execution trace and ForagerAgent data before any post-training, shaping the model's internal representations ahead of SFT and RL.
What world modeling enables — early demonstrations
The paper shows three capabilities that emerge from world model training:
Execution simulation. The model can step through Python code line by line, predicting variable states at each step, without access to an actual Python interpreter. This enables a "neural interpreter."
Reasoning via tracing. Under light prompting, CWM integrates execution trace prediction into its chain-of-thought reasoning — it thinks through a problem in natural language, then verifies its answer by simulating the execution trace, then uses the trace result to confirm or correct its output.
Reasoning-via-tracing for program generation. The model can predict execution traces for functions constrained only through natural language or formal asserts, then construct the function body from the generated execution actions — essentially deriving code from its operational semantics rather than pattern-matching from examples.
The through-line
The paper's central argument is that code understanding and code generation are currently limited by training on static text. A model that has simulated millions of Python executions — watching variables change, seeing bugs manifest in test failures, observing the full trajectory of agentic software engineering — develops a fundamentally different and deeper representation of what code means. The tracing data and ForagerAgent data are both attempts to give the model this experience at scale, ahead of the RL stage where it can then leverage that grounding to reason more effectively about code correctness and generation.

### Nemotron 3 [link] (Dec 2025)
What Nemotron 3 is
Nemotron 3 is a family of three models — Nano, Super, and Ultra — from NVIDIA, designed primarily for agentic AI applications. The central design philosophy is inference efficiency without sacrificing accuracy, achieved through a combination of architectural choices that work together rather than independently.
Hybrid Mamba-Transformer MoE architecture
The most fundamental architectural decision is replacing most attention layers with Mamba-2 layers. Rather than interleaving MoE layers with expensive self-attention layers which need to attend over a linearly increasing KV cache during generation, Nemotron 3 models predominantly interleave MoE layers with cheaper Mamba-2 layers which require storing only a constant state during generation. Only a select few attention layers are included. The three layer types serve distinct roles. MoE layers provide sparse parameter scaling — large total parameter counts at low per-token compute. Mamba-2 layers handle the bulk of sequential processing with constant memory cost regardless of sequence length. The few attention layers provide high-fidelity long-range retrieval for cases where precise token lookup genuinely matters. The throughput result is striking: Nemotron 3 Nano 30B-A3B achieves 3.3× higher throughput compared to Qwen3-30B-A3B, with further speedups at longer sequences. arxiv The "further speedups at longer sequences" is the key property — as context grows, attention cost grows linearly while Mamba cost stays constant, so the advantage compounds. The long context benefit also comes partly from not using RoPE in attention layers. Since Mamba layers provide implicit positional information, Nemotron 3 models do not use RoPE in attention layers and therefore do not suffer from out-of-distribution RoPE issues during context extension. arxiv This allowed extending to 1M token context without the staged RoPE interpolation work that pure transformer models require.
LatentMoE
Used in Super and Ultra, LatentMoE is a quality improvement at iso-cost rather than an efficiency improvement. The insight is that MoE inference bottlenecks — memory bandwidth for loading expert weights and all-to-all communication for dispatching tokens — both scale with the hidden dimension d, not with the number of experts. LatentMoE projects each token from the original hidden dimension d into a smaller latent dimension ℓ, routes to an expanded set of experts that operate entirely in this latent space, and then projects back to the original hidden dimension d. Both per-expert weight loads and communication payloads are reduced by a factor of d/ℓ. These savings are reinvested by increasing both the total number of experts from N to N·d/ℓ and the top-K active experts from K to K·d/ℓ. The bottlenecks stay approximately constant because smaller experts and more active experts cancel out. What actually changes is the nonlinear budget — the number of independent specialized transformations applied per token — which goes from 6 to 22 in the paper's example. More active experts at lower dimensionality outperforms fewer experts at full dimensionality. LatentMoE consistently outperforms the standard MoE baseline across all evaluated tasks with both models at 8 billion active and 73 billion total parameters trained for 1 trillion tokens.
Multi-Token Prediction (MTP)
MTP adds lightweight auxiliary prediction heads that predict multiple future tokens from each position during training. This provides denser supervision — each position contributes k gradient signals instead of 1 — which encourages the model to build forward-looking representations. MTP improves performance by roughly 2.4% on average across benchmarks. A lightweight MTP module achieves around 97% acceptance on the first two predicted tokens. The 97% acceptance rate comes from the heads sharing the main model's representations — they're not a separate model approximating the main model from outside, they're trained jointly on the same hidden states. This makes them highly calibrated to the main model's distribution. At inference time, the same heads that provided training signal become the draft model for speculative decoding. Because generating tokens is memory-bandwidth-bound (loading model weights dominates over computation), verifying k draft tokens in one batched forward pass costs roughly the same as generating 1 token normally. The weight loading happens once instead of k times, and the matmuls over k positions happen in parallel. This is most beneficial at batch size 1 and for long-form generation — exactly the agentic workload Nemotron 3 targets.
NVFP4 training
Super and Ultra are trained using NVFP4, a 4-bit floating point format. On GB300, peak FP4 throughput is 3× higher than FP8 throughput. The challenge with low-precision training is numerical stability. The paper's solution is selective precision: keep sensitive layers in higher precision. QKV and attention projections are kept in BF16 to maintain fidelity of the few attention layers. Mamba output projection layers have high flushes to zero up to 40% when quantized to NVFP4, so these are kept in MXFP8. The last 15% of the network is kept in high precision to maintain stability. The result: less than 1% relative difference in loss between NVFP4 and BF16 on Nano, decreasing to less than 0.6% on the larger 8B active parameter model. arxiv The loss gap shrinking with model size is consistent with broader findings in quantization research — larger models are more robust to precision reduction.
Multi-environment RL post-training
Rather than training separate RL stages for different capabilities, Nemotron 3 trains all environments simultaneously. Unlike previous models where there were separate training stages for different tasks, Nemotron 3 trains on all tasks simultaneously. Simultaneous training is more stable, less prone to reward hacking, and overall better compared to staged approaches which often result in degradation of some capabilities. The environments span mathematical and scientific reasoning, competitive coding, instruction following, software engineering, search, chat, general agentic tool use, and long context. The high inference throughput from the Mamba-Transformer architecture provides a practical advantage here — generating rollouts for RL training is cheaper when inference is faster. The paper uses GRPO with masked importance sampling to account for discrepancies between training and rollout policies — the same kind of off-policy correction we discussed throughout our RL conversations.
Reasoning budget control
Models are trained to accept a user-specified token budget for their thinking trace. When the model reaches the budget, appending the </think> token causes it to generate a response based on the partial thinking trace. This gives users a continuous accuracy-efficiency dial at inference time rather than a binary thinking/non-thinking choice.
The through-line
Every architectural decision serves the same goal: maximize accuracy per inference FLOP for agentic workloads. Mamba replaces most attention to eliminate the KV cache scaling problem. LatentMoE gets more expert diversity without paying more in bandwidth or communication. MTP improves both training signal and inference speed simultaneously. NVFP4 enables faster matmuls with negligible quality loss. And simultaneous multi-environment RL produces a more capable and stable final model than staged training. The choices compound — the 3.3× throughput advantage from Mamba makes the RL rollout generation cheaper, which enables more RL training for the same compute budget, which improves final model quality.

### GLM-5 [link] (Feb 2026)
What GLM-5 is
GLM-5 is Zhipu AI's 744B parameter MoE model (40B active, 256 experts, 8 active per token, 80 layers), doubling GLM-4.5's size. Its central thesis is the transition from "vibe coding" — ad-hoc prompt-driven code generation — to "agentic engineering" — autonomous long-horizon software development. The paper's contributions span architecture, pretraining, post-training infrastructure, and RL algorithms.
Muon Split — fixing Muon + MLA incompatibility
As we discussed in detail earlier, GLM-5 discovers that Muon's matrix orthogonalization, applied to MLA's up-projection matrices as joint matrices, degrades performance below GQA. The fix — Muon Split — splits W_{UQ}, W_{UK}, W_{UV} along the head dimension and applies Newton-Schulz orthogonalization independently per head. This restores per-head update independence, closes the performance gap with GQA, and as a bonus eliminates attention logit explosion without needing any QK-Clip strategy.
MLA-256 — hardware-aware head dimension tuning
Standard MLA from DeepSeek-V3 uses a 576-dimensional decode-time dot product, optimal for H800 GPUs but inappropriate for other hardware. GLM-5 increases head dimension from 192 to 256 and decreases head count by 1/3, keeping training FLOPs and parameter count constant while reducing decoding computation. This is a hardware-aware architectural adjustment that makes the model more efficient on non-NVIDIA chips including Chinese domestic GPUs.
MTP with parameter sharing
Rather than DeepSeek-V3's single MTP layer (trained only on 1-step prediction but used recursively for 2 steps at inference), or the expensive alternative of n separate MTP layers for n speculative steps, GLM-5 shares parameters across 3 MTP applications during training. One set of weights, unrolled 3 times, trained on 1-step, 2-step, and 3-step prediction simultaneously. Memory cost equals DeepSeek-V3's approach, but acceptance rate improves because the layer was actually trained for recursive multi-step prediction. GLM-5 achieves acceptance length 2.76 vs DeepSeek-V3.2's 2.55 at 4 speculative steps.
DSA adoption and efficient adaptation
GLM-5 adopts DeepSeek Sparse Attention through continued pretraining rather than training from scratch — avoiding the "astronomical" cost of full retraining. The two-stage process (1000-step indexer warmup + 20B token sparse adaptation) is far cheaper than DeepSeek-V3.2's 943.7B token adaptation, yet matches performance on long-context benchmarks. GLM-5 also provides an ablation study of all major efficient attention alternatives (sliding window interleave, search-based SWA, GDN, SimpleGDN), establishing that DSA is the only method that is lossless by construction — all other methods incur accuracy gaps on fine-grained retrieval.
The search-based SWA pattern discovery is itself a contribution: using beam search over layer configurations at 16K context, finding a pattern (SFSSFFSSSFFFFSSFSF...) that generalizes robustly to all tested context lengths despite being optimized only at 16K.
Three thinking modes
SFT introduces three distinct reasoning capabilities:
Interleaved Thinking — the model thinks before every response and every tool call, improving instruction following throughout agentic workflows.
Preserved Thinking — in multi-turn coding agent scenarios, all previous thinking blocks are retained across turns and reused rather than re-derived from scratch. This reduces information loss and inconsistencies when a model is maintaining plan state across dozens of tool calls.
Turn-level Thinking — per-turn control over whether the model engages extended reasoning, allowing users to disable thinking for cheap lightweight requests and enable it for complex ones within the same session.
Sequential RL pipeline with IcePop
The post-training RL is staged: Reasoning RL → Agentic RL → General RL. Each stage is specialized:
Reasoning RL uses GRPO with an IcePop-style mismatch ratio for robust on-policy optimization, explicitly suppressing samples with high training-inference divergence. Agentic RL uses asynchronous multi-task agent RL enabled by a Multi-Task Rollout Orchestrator and TITO pipelines. General RL uses hybrid rule-based, outcome, and generative reward models. Emergent Mind
IcePop's role (as we analyzed from the formula) is specifically to handle training-inference mismatch in the MoE architecture — masking tokens where the training engine (FSDP) and inference engine (vLLM/SGLang) disagree numerically due to different floating-point implementations and MoE routing sensitivity. This is distinct from DAPO's clipping which handles policy drift.
On-Policy Cross-Stage Distillation — preventing catastrophic forgetting
The sequential RL stages create a catastrophic forgetting risk — Agentic RL training can degrade Reasoning RL capabilities. The final checkpoints from all stages become teachers, and the model distills knowledge from all of them simultaneously at the end of the pipeline. Neurohive
This is a principled solution to multi-stage RL: rather than hoping each stage preserves previous capabilities (which it doesn't), explicitly distill from all stage checkpoints into one final model. The "on-policy" part means the distillation uses the current model's own rollouts rather than static teacher demonstrations, ensuring the distilled behaviors remain coherent with the model's current capabilities.
Asynchronous RL infrastructure (slime)
GLM-5 developed slime, a novel asynchronous RL infrastructure that substantially improves training throughput and efficiency by decoupling generation from training.
The key insight: synchronous RL wastes GPU time waiting for long agentic rollouts to complete. Slime's asynchronous design allows training to continue while rollouts are generated, supporting up to 1,000 concurrent rollouts for long-horizon agentic tasks. The infrastructure exposes rollout servers via HTTP APIs, allowing external agent frameworks to interact with the serving layer while the optimization backend remains unchanged.
Hierarchical Context Management for long agentic tasks
For long search and browsing tasks where context grows to 100K+ tokens, GLM-5 introduces hierarchical context management: keep only the most recent k=5 tool result rounds, and if context is still too long, clear history and continue from a clean slate. This lifted BrowseComp from 55.3% to 75.9%. 
Infrastructure: memory and parallelism
Several engineering contributions enable training a 744B MoE at scale:
Pipeline ZeRO2 gradient sharding — shards gradients across data-parallel ranks so each stage stores only 1/dp fraction of gradients, reducing persistent gradient memory via double buffering.
Zero-redundant Muon communication — restricts all-gather to parameter shards owned by each rank, eliminating redundant communication in the distributed Muon optimizer.
Pipeline activation offloading — offloads activations to host memory after forward execution, reloads before backward, overlapped with computation to achieve near-zero overhead.
INT4 QAT — quantization-aware training in the SFT stage with a unified kernel for both training and offline quantization, ensuring bitwise-identical behavior between training and inference.
The through-line
Every contribution in GLM-5 serves the same goal: make long-horizon agentic tasks tractable. DSA and MLA-256 reduce attention cost so 200K context is affordable. MTP parameter sharing improves generation speed. The sequential RL pipeline with IcePop and on-policy distillation gives the model reliable behavior across both fast reasoning and slow agentic tasks without forgetting. Slime's async infrastructure eliminates the synchronization bottleneck that would otherwise cap how complex an agentic training environment you can run. The result is a model that can genuinely reason and act autonomously for hours on complex software engineering tasks — which earlier models claimed but couldn't sustain.

### On-policy Distillation [link] (Oct 2025)
The core problem
Post-training a model requires choosing between two approaches that each have a fundamental weakness:
Reinforcement Learning (on-policy, sparse reward). The model generates its own rollouts and gets graded on the final outcome. This is on-policy — the student learns from its own mistakes in its own distribution. But the reward signal is O(1) bits per episode regardless of sequence length. A 500-token response gets one scalar: right or wrong. The model has no idea which tokens caused the failure.
SFT / Off-policy distillation (off-policy, dense reward). Train on teacher-generated examples with per-token supervision. Dense signal — O(N) bits per episode. But the student learns in states the teacher visits, not states the student visits. When the student makes an early mistake that the teacher never makes, it enters territory it was never trained on and errors compound. It can also imitate the teacher's style without learning its actual reasoning.
On-policy distillation combines both advantages: on-policy sampling (student generates its own rollouts) + dense reward (teacher grades every token).
How on-policy distillation works
The implementation is surprisingly simple:
Sample a trajectory from the student
Run the teacher on the same trajectory (one forward pass, no generation) to get teacher log-probabilities at every token
Compute the per-token reverse KL divergence as the reward:
KL(πθ∥πteacher)=log⁡πθ(xt+1∣x1..t)−log⁡πteacher(xt+1∣x1..t)\text{KL}(\pi_\theta \| \pi_\text{teacher}) = \log\pi_\theta(x_{t+1}|x_{1..t}) - \log\pi_\text{teacher}(x_{t+1}|x_{1..t})KL(πθ​∥πteacher​)=logπθ​(xt+1​∣x1..t​)−logπteacher​(xt+1​∣x1..t​)

Set the per-token advantage to negative reverse KL and train with a standard importance-sampling RL loss
The student is penalized for each token where its probability diverges from the teacher's. Tokens where the student already agrees with the teacher get no gradient. The heaviest penalties land on the "forking tokens" — the pivotal decisions where the student diverges onto a wrong reasoning path.
Crucially: the teacher never generates anything. It only scores the student's tokens. This is cheap — one teacher forward pass per rollout, not one teacher generation per rollout.
Why reverse KL specifically
The choice of reverse KL over forward KL is deliberate and has three important properties:
Mode-seeking. Reverse KL pushes the student to concentrate on one specific behavior — the teacher's mode — rather than spreading probability mass across multiple suboptimal behaviors. Forward KL is mean-seeking and would push the student to cover all of the teacher's distribution, including low-probability behaviors.
Unhackable. Unlike a learned reward model which can be gamed, low reverse KL always corresponds to genuinely high-quality behavior from the teacher's perspective. There's no reward hacking because the reward is directly grounded in the teacher's probability distribution.
Alignment with RL. Standard RL with a KL regularization term is already optimizing a form of reverse KL at the sequence level. On-policy distillation extends this to the token level — same direction, much denser signal.
Compute efficiency results
The headline numbers are striking:
Reasoning (math). Starting from the same SFT checkpoint at 60% on AIME'24:
Off-policy distillation to reach 70% requires ~2M prompts (extrapolated)
RL (Qwen3's approach) reaches 68% using ~17,920 GPU hours
On-policy distillation reaches 70% in ~1,800 GPU hours — 9-30× cheaper than either
Self-distillation experiment. Starting from scratch, train a model via RL then distill that RL-trained model back into the base model via on-policy distillation. On-policy distillation reaches the teacher's AIME performance in 7-10× fewer gradient steps, corresponding to 50-100× compute reduction when accounting for shorter context requirements and smaller batch sizes.
The information-theoretic reason: RL teaches O(1) bits per episode. On-policy distillation teaches O(N) bits per episode. For a 500-token response, that's potentially 500× more signal per rollout before accounting for gradient noise differences.
Single-prompt multi-epoch training
A surprising finding: on-policy distillation can train on the same prompt 20 times in a row with 256 rollouts each (5120 total trajectories from one prompt) and still learn the teacher's distribution rather than memorizing a single answer.
This works because the reverse KL objective pushes the student to match the teacher's full distribution — it samples different trajectories each time and gets graded on each one. With RL, training multiple epochs on the same prompt leads to memorizing the final answer. On-policy distillation learns the reasoning process rather than the answer.
This has major practical implications: collecting large diverse prompt datasets is expensive. On-policy distillation can extract much more signal from a small prompt set.
Recovering catastrophic forgetting in personalization
The second major application: a model mid-trained on domain-specific documents (internal company knowledge) loses its post-training behaviors like instruction following. This is a real and painful problem — no data mixture during mid-training fully preserves IF-eval performance, and LoRA doesn't solve it either.
On-policy distillation recovers the lost behaviors cheaply: use the original pre-midtrain model as the teacher, run on-policy distillation on general chat prompts, and the student recovers its instruction-following capability without losing the domain knowledge it just learned.
The key insight: the earlier version of the model itself serves as the reward model. You don't need a separate reward model, a human labeling pipeline, or RL infrastructure. Any instruction-tuned open-weight model's log-probabilities can be used as a reward signal. This makes on-policy distillation extremely practical for continuous learning — alternate between domain mid-training phases and distillation phases to recover behavior after each one.
Implementation simplicity
The blog emphasizes that this is a one-line change on top of an existing RL implementation that uses KL regularization — swap the regularizer model from a frozen reference copy of the student to a separate teacher model. The infrastructure is identical to standard GRPO/DAPO training.
The through-line
On-policy distillation sits in the exact position in the training taxonomy that was previously empty — on-policy sampling with dense reward. It's not novel in concept (connections to DAGGER, process reward modeling, MiniLLM), but the blog provides clean empirical evidence that it's dramatically more compute-efficient than RL for recovering or distilling capabilities, works with tiny prompt sets and multiple epochs, and is practically trivial to implement on top of existing RL infrastructure. The 50-100× compute reduction over RL for distilling a known-good policy makes it particularly valuable for the common case of domain fine-tuning where you're trying to add knowledge without losing previously-trained behaviors.

### On-policy Self-Distillation [link] (Jan 2026)
What problem OPSD solves
On-policy distillation as described in the Thinking Machines blog post we discussed earlier requires a separate teacher model — typically larger than the student — to score the student's rollouts. This is expensive and introduces a dependency on an external model. The paper asks: can a single model serve as its own teacher, eliminating the need for any external supervision?
Given that modern LLMs already exhibit strong reasoning capabilities, the core question is: can a model effectively serve as its own teacher through self-distillation? When provided with ground-truth solutions as privileged information, can a sufficiently capable model rationalize the reasoning steps and provide dense token-level supervision to guide its weaker self — the version without access to privileged information? Siyan-zhao
The central insight: privileged conditioning creates asymmetry
The key observation is that evaluation is easier than generation for LLMs. A model that can't reliably solve a hard math problem from scratch can almost certainly evaluate whether a given solution is correct and understand why it works. This asymmetry in capability can be exploited to create a teacher-student dynamic within a single model.
An LLM can be conditioned on privileged info (e.g., correct solution or a reasoning trace) and supervise its weaker self — the version without such access — by matching the privileged-info-induced distribution from itself. Rationalization — understanding a given correct answer — is similarly easier than generation, and we can utilize this to extract dense supervision from LLMs themselves. Siyan-zhao
The OPSD mechanism
A single model instantiates two roles by conditioning on different contexts:
Student policy p_S(· | x): sees only the problem x, exactly replicating inference-time conditions.
Teacher policy p_T(· | x, y*): receives the same problem plus the ground-truth solution y* as privileged context.
Both policies share identical parameters — this is not two separate models, just the same model called twice with different input contexts.
The training procedure:
The student generates a rollout ŷ from the problem alone
Both policies evaluate the student's generated trajectory at every token position — the student computes its own next-token distribution, the teacher computes a better-informed next-token distribution conditioned on knowing the correct answer
The loss minimizes the per-token divergence between the teacher and student distributions over the student's own rollouts
The divergence objective: JSD instead of pure KL
Rather than pure reverse KL (as in the Thinking Machines blog), OPSD uses the generalized Jensen-Shannon divergence:

where m = βp_T + (1-β)p_S is the interpolated mixture. This is a weighted combination of forward and reverse KL, parameterized by β. At β=1 it reduces to forward KL (teacher → student), at β=0 to reverse KL (student → teacher). JSD is symmetric and bounded, making it numerically more stable than pure KL in this setting. JSD is not a weighted mixture between forward and reverse KL in the sense of interpolating between their values. It's a single symmetric (or asymmetric) measure that uses the mixture distribution m as a common reference point for both KL terms. The β parameter controls how much the mixture is weighted toward p_T vs p_S, which in turn controls the asymmetry of the divergence — but it doesn't cleanly reduce to either pure forward or pure reverse KL at the extremes.
Per-token pointwise KL clipping — a critical engineering fix
Token-level divergence is highly skewed across vocabulary entries — a small subset of stylistic tokens such as reasoning connectives like "wait", "think", "therefore" exhibits much higher divergence than mathematically meaningful tokens. Without correction, these stylistic tokens dominate the training signal. Per-token pointwise KL clipping caps the maximum per-vocabulary-entry divergence contribution at each position. This stabilizes training and prevents performance collapse, particularly important given that OPSD converges rapidly within a few hundred steps. Siyan-zhao
This is exactly the same insight as CISPO's treatment of high-ρ tokens — stylistic fork tokens accumulate disproportionately large gradient signals and need to be bounded to prevent them from dominating training.
The through-line
OPSD sits at the intersection of several ideas we've covered. Like on-policy distillation from the Thinking Machines blog, it combines on-policy sampling with dense reward. Like CISPO, it handles high-divergence stylistic tokens specially to prevent them from dominating training. Like the zero-variance filtering in ScaleRL, it avoids the dead-gradient problem from all-correct or all-wrong batches. The novel contribution is eliminating the external teacher entirely — the ground-truth solution in the dataset becomes the privileged context that creates the teacher-student asymmetry within a single model, making the approach applicable anywhere you have a reasoning dataset with verified answers but no access to a larger teacher model.

### RL via Self-Distillation [link] (Jan 2026)
What SDPO is doing
SDPO sits in the same conceptual space as OPSD and the Thinking Machines blog — on-policy training with dense token-level supervision. But it solves a slightly different problem. While OPSD requires a dataset with verified ground-truth solutions y* to condition the teacher, and the Thinking Machines approach requires a separate larger teacher model, SDPO requires neither. It constructs the teacher entirely from environment feedback — runtime errors, compiler output, unit test logs, judge evaluations — which many verifiable environments already produce for free.
1. The core mechanism: self-teacher via feedback conditioning
The central insight is that a model that just failed at a task, when shown the feedback explaining why it failed, can recognize its own mistakes in hindsight and provide better next-token predictions than it could without that feedback. This "model conditioned on its own failure feedback" becomes the teacher.
For a rollout ŷ sampled from the student π_θ(·|x), the self-teacher is the same model run with the feedback appended: π_θ(·|x, f, ŷ_{<t}). At each token position t, the teacher's next-token distribution is better informed than the student's because it knows what went wrong. The loss minimizes the divergence between these two distributions along the student's own trajectory.
The resulting per-token advantage A_t^SDPO = log π_teacher(ŷ_t) - log π_student(ŷ_t) is positive where the teacher would endorse the sampled token and negative where it would have gone elsewhere — dense credit assignment at every token position rather than one scalar per sequence.
2. SDPO as a policy gradient algorithm — enabling drop-in replacement
The paper shows that the gradient of L_SDPO has exactly the policy gradient form. This means SDPO can be implemented as a drop-in replacement for GRPO's advantage computation in any standard RL pipeline — you swap out the scalar group-relative advantage for the per-token self-teacher advantage. Everything else (importance sampling, clipping, batch processing) stays the same. This is a significant practical contribution: adoption requires minimal engineering changes.
3. Handling scalar-only environments — implicit feedback from successful rollouts
A key extension: what if the environment only returns a binary reward (correct/incorrect) with no textual feedback? SDPO handles this by using successful rollouts as implicit feedback for failed ones. For a group of rollouts on the same prompt, correct solutions are shown as demonstrations to the self-teacher when evaluating incorrect ones. The self-teacher conditioned on "here is a correct solution" can then identify what the incorrect rollout did differently and provide corrective token-level signal.
This means SDPO strictly subsumes GRPO — it provides dense supervision even in the binary reward setting that GRPO was designed for, by reusing the group's successful rollouts rather than discarding them after computing a scalar advantage. GRPO collapses to zero gradient when all rollouts succeed or all fail (zero-variance groups). SDPO avoids this entirely because even when all rollouts are wrong, the teacher conditioned on a retrieved correct demonstration can still assign meaningful per-token advantages.
4. Three levels of credit assignment
The paper identifies a hierarchy of supervision density that SDPO can exploit:
Sequence-level (standard GRPO): one scalar reward per complete response. All tokens in a sequence get the same advantage regardless of where the error actually occurred.
Token-level (GRPO with per-token normalization, DAPO): still one scalar per sequence but normalized differently. Still no per-token credit assignment.
Logit-level (SDPO): full next-token distribution comparison at every position. The self-teacher's distribution assigns probability mass to specific tokens at specific positions, so the advantage directly identifies which tokens were mistakes and which were correct moves.
The paper empirically validates that denser credit assignment translates to faster learning — SDPO reaches GRPO's final accuracy 4-6× faster in wall-clock time.
5. Test-time self-distillation
A unique capability: SDPO can be applied at test time to a single hard problem. Given a question the model is struggling with, repeatedly sample rollouts, get environment feedback, distill the corrected teacher distribution back into the policy, and try again with the updated policy. This is gradient descent applied to a single question at inference time.
For very hard problems (pass@64 < 0.03 on LiveCodeBench), SDPO achieves the same discovery probability as best-of-k sampling with 3× fewer attempts, and discovers solutions that neither best-of-k nor multi-turn conversation can find at all. This is because binary reward RL provides zero learning signal before the first success, while SDPO can make progress from pure failure feedback — each failed attempt with a runtime error provides gradient signal even without a correct solution in sight.
6. Self-teacher quality improves during training
A subtle but important finding: the self-teacher gets better as the student improves. At the start of training, the student's policy is weak, so even the feedback-conditioned teacher can't provide great guidance. But as the student improves, it generates better partial reasoning and the teacher conditioned on feedback becomes more accurate about where things went wrong. This creates a positive feedback loop — better student → better teacher → better gradient signal → better student — that standard GRPO doesn't benefit from since its signal is always binary.
The search result confirms this: "the self-teacher improves throughout training, and the final student substantially surpasses the initial teacher."
7. Scale synergy
SDPO's gains grow with model size across Qwen3 models (0.6B → 8B), with the gap over GRPO widening as models become stronger. The hypothesis is that larger models are better in-context learners, so they can more accurately recognize their own mistakes when shown feedback, making the self-teacher more informative. This suggests SDPO's advantage over scalar-reward RL will increase as models become more capable — which is the opposite of what you'd expect from methods that rely on a fixed external teacher whose advantage shrinks as the student catches up.
The through-line
SDPO's central contribution is showing that the bottleneck in RLVR is not the lack of a stronger teacher but the underutilization of environment feedback that already exists. Runtime errors, compiler output, and judge evaluations already explain why a solution failed — these are rich natural language signals that scalar reward RL discards after computing a binary outcome. SDPO converts this discarded information into per-token gradient signal through a self-teacher that requires no external model, no dataset of demonstrations, and no reward model — just the environment feedback that verifiable domains produce naturally.

### RL via Text Feedback [link] (Feb 2026)
1. The RLTF formalization
The paper formalizes RL from Text Feedback (RLTF) as a multi-turn MDP where:
Turn 0: model generates y₀ from x₀, receives scalar reward r₀ and text feedback c₀
Turn 1: model generates y₁ from augmented prompt x₁ = f(x₀, y₀, c₀), receives reward r₁
The key distinction from standard multi-turn RL is the objective: while optimizing the multi-turn objective is straightforward, the central research question is how to design learning objectives that improve J_SingleTurn(π) — evaluating the policy on initial prompts without additional feedback at test time.
This formalization cleanly separates two things that are often conflated: getting better at multi-turn interaction (easy, naive RL does this) versus internalizing feedback to improve first-turn competence (hard, requires explicit design).
2. Self Distillation (RLTF-SD) — the first method
The core insight: text feedback is particularly valuable because it often turns an incorrect first attempt into a correct second attempt. We treat the policy acting under the post-feedback prompt as an implicit teacher, and distill it into the original one-shot policy. Distillation compiles away the need for feedback by turning test-time refinement into a training signal.
Three design choices are validated:
First-turn baseline over second-turn baseline. Using second-turn mean rewards as baseline causes gradient-signal collapse: when feedback is highly effective (p₁ → 1), all second-turn responses are correct, the centered advantages vanish, and there's no learning signal. The first-turn mean baseline avoids this — the update remains non-trivial as long as the first-turn policy is imperfect, regardless of how effective the feedback is. This is the most important algorithmic insight in section 3.
AWR over importance sampling. For LLM outputs (long token sequences), the distribution shift between first and second turn compounds across tokens, making the full importance-sampling gradient estimation high-variance. In experiments, variance dominates bias: setting πref = π(·|x₀) which removes importance weighting consistently improves stability and final performance compared to full IS correction or clipped objectives. arxiv This is a deliberate choice of mild bias over high variance, empirically validated.
Joint training with multi-turn RL. The full algorithm combines J_MultiTurn (keeping both turns strong via direct RL reward) with ℓ_distill (bridging second-turn competence into first-turn competence). Neither alone suffices — RL without distillation doesn't internalize feedback, distillation without RL allows the second-turn policy to drift.
3. Feedback Modeling (RLTF-FM) — the second method
Rather than using feedback to generate better rollouts for distillation, RLTF-FM trains the model to predict the feedback itself as an auxiliary objective.
The model learns to generate critiques as if it were the judge. This is added as an auxiliary loss alongside the standard multi-turn RL objective.
The theoretical justification is framed as a representation learning argument: reward-only learning faces two bottlenecks: rare-event estimation (when base success rate is ε₀, SNR scales as √ε₀, requiring O(1/ε₀) rollouts to estimate a single gradient component reliably) and weak identifiability of representation directions (even conditioning on success, the reward-weighted gradient concentrates on a small set of directions, leaving a nontrivial low-signal subspace that reward-only RL fails to identify early on). arxiv
Feedback modeling supplies a better-conditioned representation signal. RLTF-FM acts like a representation preconditioner: it improves the identifiability and conditioning of representation degrees of freedom that reward-only RL struggles to learn early on. arxiv
The key insight: natural language critiques are dense and structured, and predicting them forces the model to develop representations of what makes an answer wrong — representations that are useful for generating correct answers but that sparse binary reward can't identify efficiently.
4. Test-time scaling via self-feedback
A unique emergent capability of RLTF-FM: because pθ(c | x, y) is produced by the same policy, the model can be run in feedback mode at inference time to generate critiques and perform iterative refinement: sample y₀, generate c̃₀ from the model itself, update x₁, resample y₁. This enables test-time scaling without requiring a separate learned judge model. arxiv
The model learns to critique during training and this transfers to useful self-critique at inference time. Standard RL with self-critique training alone doesn't achieve this — the feedback modeling loss is what makes the self-critiques actually informative enough to drive test-time improvement.
5. Empirical findings
Both RLTF-SD and RLTF-FM consistently outperform all baselines across tasks. Notably, GRPO with multi-turn training demonstrates similar single-turn performance to single-turn training, suggesting that naively incorporating feedback as additional context is insufficient to internalize its learning signal. arxiv
The gains are particularly large on reasoning tasks — Knights and Knaves improves from 0.373 (standard GRPO) to 0.802 (RLTF-SD) and 0.880 (RLTF-FM). Math gains are more modest but consistent.
Semantic richness matters. Replacing full text critiques with correctness-only feedback ("your previous answer was correct/incorrect") significantly degrades performance, confirming that the information density of natural language feedback is doing real work, not just acting as a binary signal.
The through-line
The paper's central contribution is identifying and solving the feedback internalization problem: multi-turn RL with feedback improves multi-turn performance but not single-turn performance because the policy learns to use feedback as context rather than learning from it. The two methods fix this from complementary angles — RLTF-SD directly distills feedback-corrected behavior into the no-feedback policy, while RLTF-FM builds internal representations of failure modes that improve generation even without external feedback. Together they expand what RL can accomplish to settings where sparse binary reward provides insufficient learning signal — tasks where the model is too weak to generate correct solutions regularly, making standard GRPO's zero-variance collapse problem acute.

This is the most important scientific contribution and connects directly to everything we've discussed about GRPO, DAPO, Dr. GRPO, and CISPO. Not all recipes yield similar asymptotic performance. Details such as loss aggregation, normalization, curriculum, and off-policy algorithm primarily modulate compute efficiency without materially shifting the asymptote. Stable, scalable recipes follow predictable scaling trajectories. Translated: some design choices raise the ceiling (A) — these are the ones that really matter at large scale. Other choices only affect how quickly you reach the ceiling (B) — they look important at small compute but become irrelevant once you have enough budget. Methods that appear superior at small compute budgets can be worse when extrapolated to large-compute regimes. We can still identify scalable methods by estimating the scaling parameters from early training dynamics. This is the "bitter lesson" framing — the right answer at small scale is often wrong at large scale.



The most important design decisions
Through systematic ablation, three decisions stand out as affecting the asymptotic ceiling A.
Loss type. Both GSPO and CISPO substantially outperform DAPO, improving the asymptotic pass-rate A by a large margin. CISPO exhibits a prolonged near-linear reward increase and is marginally better than GSPO later in training. arxiv This is a direct empirical confirmation of what we discussed — CISPO's preserved gradient signal for all tokens matters at large compute scale.
FP32 precision at the LM head. Generators and trainers rely on different kernels for inference and training, leading to small numerical mismatches in token probabilities. The precision fix dramatically improves asymptotic performance A from 0.52 to 0.61. arxiv A 9 percentage point gain from a precision fix alone — this is the training-inference logit mismatch problem that also motivated the Nemotron 3 and K2.5 papers.
PipelineRL vs PPO-off-policy. PipelineRL and PPO-off-policy achieve similar asymptotic performance A, but PipelineRL substantially improves compute efficiency B, reaching the ceiling faster. This is because PipelineRL reduces idle time in the training process. arxiv

### How to Scale Your Model [link]
Chapter 1: Rooflines [link]
The practical takeaways
The page distills to a few rules practitioners should internalize:
Rule 1. For a bf16 matmul, batch size > 240 tokens per replica → compute-bound. Below 240 → bandwidth-bound and you're wasting hardware FLOPs.
Rule 2. Int8 weight quantization halves the critical batch size to ~120.
Rule 3. Elementwise operations (activations, layer norm) are always bandwidth-bound regardless of batch size — their arithmetic intensity is constant and low.
Rule 4. For multi-chip sharded matmuls, the bottleneck shifts from HBM bandwidth to ICI bandwidth, and the critical threshold becomes a function of model dimension rather than batch size.
Rule 5. Lower and upper bounds on runtime differ by at most 2× when assuming perfect overlap of compute and communication — a useful sanity check for back-of-envelope estimates.
Chapter 2: TPUs [link]
Practical rules from the page
Rule 1: Weight matrices must be padded to multiples of 128. Weight matrices need to be padded to at least size 128 (256 on TPU v6e) in both dimensions to fill up the MXU. Smaller axes are padded to 128. github If your model dimension is not a multiple of 128, you're wasting MXU capacity.
Rule 2: Lower precision is faster. Lower precision matrix multiplication tends to be faster. TPUs can do int8 or int4 FLOPs roughly 2x/4x faster than bfloat16 FLOPs for generations that support it. github
Rule 3: VMEM prefetching can hide HBM latency. VMEM bandwidth is around 22x higher than HBM bandwidth which means an MXU operation reading from VMEM requires an arithmetic intensity of only 10-20 to achieve peak FLOPs utilization. In a normal Transformer we can sometimes load feed-forward weights into VMEM during attention, which can hide the cost of the weight load if we're memory bandwidth bound. github If weights fit in VMEM, the critical batch size drops from ~240 to ~10-20.
Rule 4: ICI topology matters — only nearest neighbors are directly connected. Within a slice, TPUs are only connected to their nearest neighbors via ICI. This means communication over ICI between distant chips in a slice needs to hop over the intervening chips first. github Data traveling across a large pod hops through intermediate chips, each hop adding latency.
Rule 5: Toroidal wraparound cuts maximum distance in half. The toroidal structure reduces the maximum distance between any two nodes from N to N/2, making communication much faster. github But you only get wraparound when axes are full size (16 for v5e/v6e, multiples of 4 for v4p/v5p) — smaller slices lose this benefit and roughly double communication time.
Rule 6: Communication must be proportional to bandwidth at each level. To avoid bottlenecking the TPU compute unit, we need to make sure the amount of communication across each channel is proportional to its speed. github The bandwidth hierarchy in order: HBM (2.8 TB/s) → ICI (90 GB/s) → PCIe (16 GB/s) → DCN (6.25 GB/s). Tensor parallelism belongs on ICI. Data parallelism can tolerate DCN.
Rule 7: PCIe is roughly 100x slower than HBM. PCIe bandwidth is ~16 GB/s vs HBM at ~1-2 TB/s. Weight offloading to CPU DRAM is extremely expensive — the question 3 answer on the TPU page showed a critical batch size of 61,000 tokens just to be compute-bound over PCIe. In practice, offloading to host memory is almost always the bottleneck.
Rule 8: DCN requires four hops and is the slowest path. To transfer buffers from TPU to TPU over DCN, we first need to transfer over PCIe to the host, then egress over the network, then ingress over the target host network, then over PCIe into HBM. github At 6.25 GB/s this is so slow that any communication-heavy parallelism (tensor parallelism) must stay within a single ICI-connected slice, never crossing DCN.
Rule 9: A full TPU v5p pod can do 4 exaFLOPs/s. A single pod of 8960 chips can do 4 exaflops per second. That's one of the most powerful supercomputers in the world. And Google has a lot of them. github
Rule 10: Single-chip inference latency is lower-bounded by weight load time. The worked problem on the page shows that loading a 200B parameter model in BF16 across 32 TPU v4p chips takes at minimum ~10ms per token — and this is actually close to achievable in practice at small batch sizes since decoding is memory-bandwidth-bound.

### Compute Optimal RL LLM Scaling [link] (2026)
What the paper is asking
The central question is: given a fixed compute budget for RL post-training, how should you allocate it across the three knobs that control sampling?
Total rollout compute C scales as B_problem × n × M, where n is the number of parallel rollouts per problem (group size in GRPO), B_problem is the number of unique problems per batch, and M is the number of sequential gradient update iterations. Given a base model, a problem set, and a sampling budget, the paper asks: find the configurations of n, M, and B_problem that attain the best possible performance.
This is analogous to Chinchilla-style scaling laws for pretraining (how to split compute between model size and tokens), but for RL post-training.
Why RL scaling laws are harder to derive than pretraining scaling laws
In supervised learning, the objective is smooth and optimized over a fixed data distribution, resulting in predictable optimization dynamics. In RL, two challenges arise: the expected-reward objective behaves non-smoothly across training iterations, and the data distribution itself depends on the policy being optimized. In online on-policy RL, the base policy and every policy update determines the distribution of future experience, which is non-stationary — violating the stationary data assumption that underlies standard scaling law intuitions.
Their solution is to first find a "healthy" RL recipe where training is stable, and only then fit scaling laws on top of it.
What makes a healthy RL recipe — three factors
Factor 1: Dataset difficulty. Easy problems (where the model already gets them right often) tend to produce rapid entropy collapse. Hard problems (where the model rarely produces any correct trace) require careful optimization that actively couples with exploration. Running standard RL recipes on easy problems often results in instability when the problem set is too hard.
Factor 2: Entropy and KL regularization — and the key finding here is counterintuitive. On easy problems, running RL without a sufficiently large entropy regularizer often leads to premature entropy collapse. However, on hard problems, running RL with an entropy regularizer alone often leads to an explosion in entropy and response length. Removing KL and entropy regularization entirely yields the most stable training dynamics on hard problems. The paper therefore adopts KL+entropy regularization on the easy set and no KL or entropy regularization on the hard set. This directly contradicts the common practice of always using KL regularization. On hard problems where the model needs to explore, the KL anchor becomes an obstacle.
Regarding zero-variance filtering: applying it only to the KL and entropy terms (not the policy gradient itself) mitigates some instability on hard problems, but does not fully eliminate it. The most stable configuration on hard problems removes regularization entirely.
Factor 3: Learning rate scaling. The learning rate should not be fixed as total batch size B changes. Square-root scaling (η ∝ √B) enables faster convergence while avoiding the instability seen in linear scaling. A base learning rate of 1e-6 at B=1024 is identified as the anchor. 
The main scaling findings
Finding 1: Optimal n grows with compute budget, following a sigmoid. As the total sampling budget C increases, the optimal number of parallel rollouts per problem n grows following a predictable sigmoid. To maximize performance, practitioners should configure n based on their compute budget: allocating larger n at higher compute budgets and smaller n at lower budgets. In GRPO terms: if you have more compute, you should increase group size. If you have limited compute, keep group size small and use more sequential iterations instead.
Finding 2: Easy and hard problems benefit from more rollouts via different mechanisms. On easy problems, larger n primarily sharpens the policy on already-solvable problems — the improvement shows up in the worst@k metric (making the model more reliably correct). On hard problems, scaling n is essential for expanding coverage — discovering rare successful trajectories — directly improving the best@k metric. This is intuitive: on easy problems you have plenty of correct examples and need to consolidate; on hard problems you need more samples just to find any correct solution at all.
Finding 3: B_problem has marginal impact within a moderate range, but drives a dynamic allocation rule. Varying the number of unique problems per batch yields negligible changes in validation reward, provided it is neither too small nor too large, given a fixed n. Under fixed hardware constraints on total batch size (B = B_problem × n), this insensitivity dictates a dynamic allocation strategy: prioritize large B_problem (small n) at low compute budgets, but shift to large n (small B_problem) at high compute budgets to maximize performance. In other words: at low compute, diversity of problems matters more than depth per problem. At high compute, depth (more rollouts per problem) matters more than breadth.
Finding 4: The saturation point of n is context-dependent. The benefits of scaling n hold across different base models and datasets, but the specific value at which n saturates depends on: base model capacity (stronger models tolerate larger n before plateauing), problem set size (smaller datasets lower the saturation point due to early overfitting on easy problems), and problem distribution (harder problems saturate at lower n because they require a critical mass of sequential updates, constraining the budget available for width).
The practical takeaway — the IsoCompute Playbook
The paper is prescribing a concrete workflow:
Before running RL, measure your base model's pass rate on your problem set to classify it as easy or hard
Choose your regularization regime accordingly — KL+entropy for easy, none for hard
Use square-root LR scaling as you scale batch size
At low compute budgets, use more unique problems per batch and fewer rollouts per problem
At high compute budgets, flip the ratio — fewer unique problems but more rollouts per problem
The optimal n grows predictably as a sigmoid of compute, so you can extrapolate from small experiments
The deeper insight is that RL scaling laws are not universal — they're governed by the interaction between compute, the base model's current capability, and the difficulty distribution of the training data. The paper provides a framework for thinking through this interaction systematically rather than tuning hyperparameters by hand.
Glossary of Terms
FSDP — training framework (or Megatron)
FSDP (Fully Sharded Data Parallelism) is a PyTorch feature for training large models across many GPUs. Its job is distributing model parameters, gradients, and optimizer states across GPU memory so that models that don't fit on one GPU can be trained.
Sharding. FSDP splits (shards) the model's parameters across all GPUs in a data-parallel group. Each GPU only stores a fraction of the full model weights. When a layer needs to be computed, an all-gather operation reassembles the full weights temporarily, then discards them after.
Gradient accumulation. During the backward pass, FSDP manages gradient computation and reduction across GPUs.
Optimizer states. Adam's m and v moment estimates are also sharded — each GPU only stores the optimizer state for the parameters it owns.
FSDP uses PyTorch’s standard operators, full precision, different memory layouts. FSDP has no concept of fast autoregressive generation — it's designed for the training loop (forward pass → loss → backward pass → optimizer step), not for generating thousands of tokens efficiently.
vLLM — inference engine (or SGLang)
vLLM is optimized for generating text as fast as possible. Its key contributions:
PagedAttention. Instead of allocating a contiguous KV cache for each sequence (which wastes memory due to fragmentation), vLLM manages the KV cache like virtual memory — dividing it into fixed-size pages that can be allocated non-contiguously. This allows much higher GPU utilization when serving many requests simultaneously.
Continuous batching. Rather than waiting for all sequences in a batch to finish before starting new ones, vLLM adds new requests to the batch dynamically as slots free up. This keeps the GPU busy rather than idling while long sequences finish.
Optimized kernels. vLLM uses highly optimized CUDA kernels specifically for the autoregressive decode step — the memory-bandwidth-bound operation of loading weights and generating one token at a time.
vLLM uses FlashAttention kernels, custom CUDA code, potentially quantized weights. vLLM does not compute gradients. It has no optimizer, no backward pass, no gradient accumulation. It's purely forward pass for generation.

## Details

### 5–8. The RL-Objective Family (grouped for direct comparison)

Everything here is a variant of **GRPO** (Group Relative Policy Optimization, from DeepSeekMath / DeepSeek-R1, arXiv:2402.03300 / 2501.12948). GRPO drops PPO's value network and instead samples a **group** of G responses per prompt, computes each response's reward, and sets the advantage to the **group-normalized reward**: Â = (r − mean(r_group)) / std(r_group). It keeps PPO's token-level clipped importance-sampling surrogate. This makes it cheap (no critic) but introduces the biases and instabilities the next four papers fix.

#### DAPO (arXiv:2503.14476, ByteDance Seed + Tsinghua AIR)
"Decoupled Clip and Dynamic sAmpling Policy Optimization." Per the paper it "achieves 50 points on AIME 2024 based on Qwen2.5-32B, outperforming DeepSeek-R1-Zero-Qwen-32B (47 points) using 50% training steps" — while a **vanilla GRPO baseline reaches only 30 points**; it is open-sourced on the **verl** framework and trained on 128 H20 GPUs. Four techniques, each fixing a specific failure:
1. **Clip-Higher** (asymmetric ε): decouples the lower and upper clip bounds, ε_low = 0.2, ε_high = 0.28. Fixes **entropy collapse** — with symmetric PPO clipping, already-likely tokens can't grow enough while low-prob tokens get suppressed, so the policy prematurely narrows. Raising the upper bound lets low-prob "exploration" tokens increase.
2. **Dynamic Sampling**: oversample and **filter out prompt groups where all responses are correct (acc = 1) or all wrong (acc = 0)**, because those yield zero advantage → zero gradient. Keeps every batch full of effective gradients; improves efficiency and stability.
3. **Token-level Policy Gradient Loss**: sum the loss over all tokens in the batch and normalize by the **total token count**, not by averaging per-response then per-group. Under group-mean normalization, long responses' tokens get down-weighted, so a repetitive/garbage long generation is under-penalized. Token-level normalization fixes this.
4. **Overlong Reward Shaping**: soft length penalty ramping in before max length, to avoid noisy reward on truncated-but-correct answers.

#### Dr. GRPO (arXiv:2503.20783, "Understanding R1-Zero-Like Training", Sea AI Lab / NUS)
Identifies and removes **two biases in vanilla GRPO's optimization**:
- **Length-normalization bias**: GRPO divides each response's loss by its own length |o_i|, making per-token gradient magnitude depend on length. Because a *wrong* answer gets a negative advantage divided by its length, longer wrong answers are penalized *less per token* → the optimizer inflates the length of incorrect responses over training. This is the notorious "response length keeps growing" artifact.
- **Std-normalization (advantage) bias**: dividing the advantage by std(r_group) over-weights question-groups with low reward variance (very easy or very hard questions), biasing the update.
- **Fix**: drop the per-response length division (use a global constant / group-mean length as uniform scaling) and drop the std division → unbiased gradient, "GRPO Done Right." Empirically cuts incorrect-response length substantially while preserving accuracy; per the paper, RL-tuning **Qwen2.5-Math-7B on MATH level 3–5 reached SOTA in only 27 hours on 8×A100, hitting 43.3% on AIME 2024 with a 7B base model.**

#### CISPO (arXiv:2506.13585, MiniMax-M1)
"Clipped IS-weight Policy Optimization." Mechanism: **clip the importance-sampling weight, not the token's gradient.** In PPO/GRPO/DAPO, when a token's ratio r_t leaves the clip band, the min(·)/clip(·) surrogate becomes *constant in θ* → that token contributes **zero gradient.** MiniMax shows these clipped-away tokens are disproportionately the **low-probability "fork" tokens** ("However", "Recheck", "Wait", "Aha") that steer reasoning branches — exactly the ones reasoning RL needs to reinforce. CISPO instead:
- Uses a REINFORCE-style objective where the IS weight is **clipped as a detached (stop-gradient) coefficient** — `clamp(exp(logratio), max=ε_high).detach()` × advantage × token log-prob.
- **No token is ever masked**: gradient flows through log π_θ for every token; only the *magnitude* (variance) of the weight is bounded.
- Canonically **single-sided** (recommended ε_clip_higher ≈ 4.0, far larger than PPO's ~1.2). Abandons the trust-region / two-sided clip entirely.
- Enabled MiniMax-M1's **full RL training on 512 H800 GPUs in three weeks, at a rental cost of just $534,700**; on a controlled Qwen2.5-32B study, **CISPO achieves a 2× speedup vs DAPO.** ScaleRL (arXiv:2510.13786) adopts the same surrogate.

#### GSPO (arXiv:2507.18071, Qwen team)
"Group Sequence Policy Optimization." Argues GRPO **misuses importance sampling**: it applies a per-token ratio using only a single sample per token, which is high-variance and, crucially, **catastrophic for MoE.** Concretely, per the paper, "with the 48-layer Qwen3-30B-A3B-Base model, after each RL gradient update and for the same rollout sample, roughly 10% of the experts activated under the new policy differ from those under the old policy" → token-level ratios become wildly noisy and can cause irreversible collapse. GSPO's fix:
- Define the importance ratio at the **sequence level**: s_i(θ) = (π_θ(y_i|x)/π_old(y_i|x))^(1/|y_i|), length-normalized to reduce variance.
- **Clip whole sequences**, not tokens. Clip range in the paper: 3e-4 (low) / 4e-4 (high) — tiny because they act on the sequence-normalized ratio.
- Result: stabilizes MoE RL without hacks like "Routing Replay"; underpins Qwen3. Trade-off: uniform per-sequence weighting can slightly hurt dense models and discards intra-sequence token structure (motivating follow-ups like ESPO/LUSPO).

### 1. GLM-5 (arXiv:2602.15763, Zhipu AI / Tsinghua) — "from Vibe Coding to Agentic Engineering"
**Lead facts**: 744B total / 40B active MoE (doubles GLM-4.5's 355B/32B), 256 experts, 80 layers (reduced to cut expert-parallel comms), sparsity dropped to ~5.4%. Adopts DSA. slime async RL infra.

- **Muon Split** (the optimizer fix, and the highest-value talking point): with the Muon optimizer, MLA (576-dim latent KV) *underperformed* GQA-8 (2048-dim KV). Root cause: Muon applies matrix orthogonalization to the whole up-projection matrix, forcing all attention heads to update at the same scale — which fights the very premise of multi-head attention. **Muon Split splits the projection matrix into per-head submatrices and orthogonalizes each independently**, letting heads update at different scales. This closed the MLA↔GQA gap (e.g., C-Eval 59.7→62.1, BBH 48.9→51.8 in the ablation) and — bonus — kept attention logits stable during pretraining **without any clipping strategy** (a direct contrast with Kimi's QK-Clip). This is the concrete answer to "how do labs configure the optimizer differently": GLM adapts orthogonalization *granularity* rather than resetting states.
- **MLA-256** (hardware-aware head dim): MLA's decode does a 576-dim dot product vs GQA's 128-dim — expensive on non-H800 hardware (DeepSeek-V3's head count was tuned to the H800 roofline). GLM raises head dim 192→256 and **cuts head count by ~1/3**, keeping training FLOPs and params constant while lowering decode compute. Deployment-efficiency-driven (matters for Ascend / Chinese chips, where §5 details W4A8 mixed-precision).
- **MTP with parameter sharing**: DeepSeek-V3 trains 1 MTP layer but predicts 2 tokens at inference — the train/infer mismatch lowers 2nd-token acceptance. GLM-5 **shares parameters across 3 unrolled MTP applications**, matching DeepSeek-V3's draft-model memory footprint while raising acceptance length (longer than DeepSeek-V3.2 at equal speculative steps on their private prompt set).
- **DSA via continued pretraining**: DSA (from DeepSeek-V3.2) is content-based sparse attention — a lightning indexer scores past tokens, top-k selected, attention only over those. Attractive because you can add it to an existing base rather than train from scratch. GLM starts from the post-mid-training base: 1000-step warmup (14 sequences of 202,752 tokens each, LR 5e-3), then a sparse-adaptation stage reusing mid-training data/hparams for 20B tokens. Cuts long-context attention cost ~1.5–2×.
- **Three thinking modes**: Interleaved (think before every response/tool call), Preserved (maintain reasoning history across agentic turns), Turn-level.
- **Sequential RL pipeline**: **Reasoning RL → Agentic RL → General RL.** Backbone = **GRPO + IcePop.** IcePop mitigates training-inference mismatch via **token-wise masking + selective gradient clipping on the ratio between the training and inference engines' log-probs** — discard/mask tokens where the two engines diverge too much (essential for MoE at scale; originated with Ring-1T). Erroneous trajectory segments are retained but **masked out of the loss** so the model learns error-correction without reinforcing the error. Agentic RL uses async multi-task rollouts with **TITO (Token-In-Token-Out)** pipelines and **direct double-sided importance sampling** for token clipping, plus dropping off-policy/noisy samples and DP-aware routing.
- **On-policy cross-stage distillation**: to prevent catastrophic forgetting across the three sequential RL stages, distill on-policy from the previous stage's model into the current one.
- **slime infra**: decouples generation from training; scales to a large number of concurrent rollouts; heartbeat-driven fault tolerance; tail-latency optimization for RL rollouts.

### 2. Kimi K2.5 (arXiv:2602.02276, Moonshot AI) + full K1.5→K2→K2.5 evolution
**Lead facts**: built on the Kimi K2 base (1.04T MoE, 32B active, 384 experts, 8 activated per token, sparsity 48, MuonClip optimizer). Adds native multimodality (MoonViT-3D), Agent Swarm/PARL, Toggle, and a novel RL clip. §4.4.2's RL algorithm is the shared foundation for both text-vision joint RL and PARL.

- **The RL objective (Eq. 1)** — a hybrid of three lineages:
  1. **K1.5's online mirror descent** with an **L2 log-ratio (relative-entropy) regularizer** −τ·(log(π_θ/π_old))².
  2. **DAPO-style token normalization**: the whole loss is divided by **N = total generated tokens in the batch** (sum over all K responses and all their tokens, not group mean). Advantage = r(x,y_j) − mean group reward r̄(x) (GRPO-style leave-in baseline).
  3. **A novel sign-agnostic gradient-masking clip**: gradients are computed normally for tokens whose ratio π_θ/π_old ∈ [α,β] and **zeroed for tokens outside the band — regardless of the advantage's sign.** The paper states the key distinction from PPO explicitly: "our method relies strictly on the log-ratio to explicitly bound off-policy drift, regardless of the sign of the advantages." This differs from PPO/GRPO/DAPO (clip conditioned on advantage sign) and from CISPO (which never zeros gradient). α, β, τ are positive hyperparameters (values not disclosed). The RL objective is minimized with the **MuonClip optimizer.** The paper frames this as departing from K1.5 to mitigate off-policy divergence "amplified by discrepancies between training and inference frameworks" (citing the Yao et al. off-policy-framework and IcePop papers) — the same mismatch theme as IcePop/GSPO; they call it "essential for maintaining training stability in complex domains requiring long-horizon, multi-step tool-use reasoning."
- **Toggle** (token-efficient RL): alternates every m iterations between **Phase0 (budget-limited)** and **Phase1 (standard scaling).** Phase0 applies a per-problem token budget — the ρ-th percentile of correct-response lengths, estimated once at start and fixed — but **only enforces it when the model's mean accuracy on that problem exceeds threshold λ** (so it never sacrifices quality on hard problems: reward is retained if mean accuracy < λ OR response length ≤ budget). Phase1 lets the model generate to max length to preserve inference-time scaling. Framed as a **stochastic alternating optimization of a bi-objective problem**, it solves the **length-overfitting** failure of rigid budgets (where budget-trained models default to truncated reasoning and can't use more compute at test time — the pathology of K1.5's continuous length penalty). Evaluated on Kimi K2 Thinking, **Toggle cuts output tokens 25–30% with negligible performance impact**, and generalizes out-of-domain (math/code training still reduces tokens on GPQA and MMLU-Pro).
- **Agent Swarm / PARL**: a trainable **orchestrator** dynamically decomposes tasks and spawns up to 100 **frozen** sub-agents running in parallel (up to ~1,500 coordinated steps). Only the orchestrator is RL-trained; sub-agent trajectories are excluded from the objective → avoids credit-assignment ambiguity and instability. Reward is shaped to encourage parallelism early then shift toward end-to-end quality (prevents "fake parallelism"), and the **critical-steps metric** makes optimization latency-aware (rewards shortening the slowest execution path, not merely spawning agents, which would inflate coordination overhead). Whether/when/how to parallelize is *learned*, not presumed. Results: BrowseComp 60.6→78.4% in swarm mode; up to 4.5× execution efficiency / ~80% runtime reduction.
- **Joint text+vision thesis**: native multimodal pretraining (~15T vision-text tokens on top of K2), **MoonViT-3D** unified image+video encoder (early fusion, shared embedding space; ~400M params), and **zero-vision SFT** (text-only SFT activates visual reasoning). Visual RL is shown to improve *text* performance too. A Decoupled Encoder Process (DEP) handles load imbalance from variable-size visual inputs.
- **Evolution table**:
| | K1.5 (Jan 2025, 2501.12599) | K2 (Jul 2025, 2507.20534) | K2.5 (Feb 2026, 2602.02276) |
|---|---|---|---|
| RL algo | online mirror descent, KL-regularized, **no value net, no MCTS** | K1.5-derived KL-regularized PG + self-critique-rubric rewards | K1.5 OMD (τ log-ratio²) + DAPO total-token norm + **novel sign-agnostic gradient-mask clip** |
| Optimizer | AdamW-class | **MuonClip (Muon + QK-Clip)** | MuonClip (inherited) |
| Length/efficiency | length penalty + long2short distillation | budget approach | **Toggle** (alternating budget / scaling) |
| Headline | long-CoT RL, multimodal, partial rollouts | agentic data synthesis, 1T MoE | native multimodal, Agent Swarm/PARL |

### 3. Nemotron 3 (arXiv:2512.20856, NVIDIA; Super variant arXiv:2604.12374)
**Lead facts**: hybrid **Mamba-2 + Transformer + MoE**; Super = 120B total / 12B active; pre-trained in **NVFP4** on 25T tokens; up to 1M context; up to 2.2× throughput vs GPT-OSS-120B and 7.5× vs Qwen3.5-122B.
- **Hybrid architecture**: mostly Mamba-2 layers (linear-time sequence modeling) with strategic global-attention layers + MoE FFNs — best-in-class throughput at on-par/better accuracy than dense Transformers.
- **LatentMoE**: a novel hardware-aware expert design optimizing **accuracy per FLOP and per parameter** simultaneously (beats standard granular MoE); first used in Super/Ultra.
- **MTP** doubles as (a) a training signal that modestly improves quality and (b) the **draft model for native speculative decoding.**
- **NVFP4 with selective precision**: the majority of floating-point multiply-accumulates run in 4-bit NVFP4 from the first gradient step (native low-precision, not post-training quant), so the model learns to be accurate within 4-bit arithmetic. Sensitive ops kept higher: **BF16 for QKV/attention, MXFP8 for Mamba output projections.** ~4× inference speedup on B200 vs FP8 on H100.
- **RL**: **GRPO with masked importance sampling** to handle the training/rollout policy discrepancy (the mismatch theme again); **multi-environment simultaneous RL training**; granular **reasoning budget control.** Uses the NeMo-RL / "NeMo Gym" environment stack.

### 4. CWM / Code World Model (arXiv:2510.02387, Meta FAIR) — RL recipe emphasized
**Lead facts**: 32B dense decoder-only, 64 layers, GQA (48Q/8KV), SwiGLU, RMSNorm, scaled RoPE; **sliding-window attention alternating local-8k and global-131k blocks in a 3:1 ratio**, 131k effective context. Four stages: pretrain (8T, code-heavy, 8k ctx) → mid-train (+5T, 131k ctx) → SFT (100B, reasoning format) → **multi-task RL (~172B tokens at 131k).**
- **Python execution tracing data**: observation-action trajectories where the model sees code and predicts how each executed line changes local-variable state (function-level, CodeContests, repository-level, and natural-language tracing variants). Teaches execution *semantics*, not just syntax — the world-model premise.
- **ForagerAgent** mid-training data: ~3M synthetic agentic bug-fix trajectories across ~10k executable repo images, covering **mutate-fix** (inject a bug, then fix it) vs **issue-fix** (fix from an error message) tasks.
- **Executable repo images infra**: RepoAgent + Activ pipelines build containerized, runnable repository environments the ForagerAgent forages in.
- **THE RL RECIPE (flagship)** — a **multi-turn GRPO variant** (paper §C). Deviations from vanilla GRPO:
  - **Multi-turn**: sequences contain both model- and environment-generated tokens (a POMDP; environment tokens are masked from the loss).
  - **No σ-normalization**: drops GRPO's std division (the same bias Dr.GRPO flags) — advantage is centered but not scaled.
  - **Asynchronous**: decouples generation from training for throughput, vs GRPO's synchronous gen↔train alternation.
  - **Four RL task types in one joint phase**: (1) Agentic SWE (SWE-bench-style), (2) Coding (CodeContests-style competitive problems), (3) Agentic coding, (4) Mathematics. Each = dataset + verifier (unit tests) + environment. Minimal toolset: **bash / edit / create / submit.** Outcome rewards (tests pass / answer correct). A **bootstrapping loop** feeds high-quality RL trajectories back into SFT, reinforcing successful patterns.
  - Infra: FlashAttention-3, FSDP + TP, FP8. No RLHF (research-only model). Per the paper, CWM reaches **65.8% pass@1 on SWE-bench Verified with test-time scaling and 53.9% without (avg of 4 runs), rising to 80.4% pass@k at k=40**, plus 68.6% LiveCodeBench, 96.6% Math-500, 76.0% AIME 2024.

### 9. slime (THUDM; github.com/THUDM/slime; lmsys.org/blog/2025-07-09-slime)
**Lead facts**: an SGLang-native RL post-training framework; the RL infra behind GLM-4.5 through GLM-5.2. Philosophy: don't fork frameworks per task — route everything through one training / rollout / data-buffer path.
- **Three modules**: (1) **Training** = Megatron-LM (actor, optional critic, weight sync to rollout, all Megatron parallelisms TP/PP/EP/CP); (2) **Rollout** = SGLang + sgl-router (generation, reward/verifier output, multi-turn/tool support); (3) **Data Buffer** = bridge holding prompts, partial rollouts, and scored samples between iterations.
- **Decoupling generation from training**: two modes — **synchronous/colocated** (share GPUs, alternate gen and train; lowers the hardware bar) and **asynchronous/decoupled** (separate GPU clusters per engine; higher throughput). The rollout server exposes a **single HTTP, OpenAI-compatible endpoint** via sgl-router, so environments interact uniformly and train/deploy stay consistent.
- **Why async beats synchronous (TRL/verl)**: in synchronous frameworks GPUs idle waiting on the slowest rollout; slime's decoupled design + partial-rollout (APRIL-style, `--partial-rollout`, `--over-sampling-batch-size`) removes "bubbles," lifting throughput (reported 9.3k→13k tok/s, +40%, at 16k max response length). Choosing SGLang as the *single* rollout backend lets it use SGLang-specific weight-sync (bucketed MoE parameter updates, `/abort_request` for dynamic sampling like DAPO), caching, and PD-disaggregation directly rather than a lowest-common-denominator multi-backend abstraction. Native Megatron/SGLang arg pass-through (`--sglang-` prefix) means it inherits upstream optimizations without wrapper code. Large MoE recipes commonly run **BF16 training state + FP8 rollout.**

### 10. Jax Scaling Book (jax-ml.github.io/scaling-book)
- **Ch. 3 "Sharding Matrices"**: builds a mental model for matmuls on sharded arrays using named-axis (block-matrix) notation. Sharded axes are partitioned across mesh axes; any unused mesh axis ⇒ the array is replicated along it. The three collectives that arise: **AllReduce** (sum partial results, everyone gets the full result — costs ~2× an AllGather), **ReduceScatter** (sum but leave the output sharded — about half the cost of AllReduce), and **AllGather** (materialize a sharded axis). Key rule for stacking two FFN layers without resharding: combine an **AllGather at the start with a ReduceScatter at the end** instead of a full AllReduce, and overlap the AG with the first matmul and the RS with the second. This is the notation used to reason about **DP / TP / PP / EP** trade-offs: **TP** shards the feature/head dim (needs per-layer comms), **DP** shards the batch (comms only at gradient sync), **EP** shards experts (all-to-all routing), **PP** shards layers (introduces pipeline bubbles).
- **Ch. 4 "Transformers"**: parameter and FLOPs counting. Rules of thumb: **params ≈ 12·n_layers·d_model²** for a standard dense transformer (attn + MLP); **training FLOPs ≈ 6·N·D** (N params, D tokens) — forward ~2N per token, backward ~4N. **KV cache ≈ 2·n_layers·n_kv_heads·d_head·seq_len·batch·bytes** — the dominant inference-memory term at long context, which is the motivation for MLA/GQA/DSA/TurboQuant. **Attention vs MLP FLOPs**: per-token MLP FLOPs are roughly constant in sequence length while attention FLOPs grow with sequence length — so MLP dominates at short context and attention overtakes it once the sequence length becomes large relative to d_model. Chs. 1 (Rooflines) and 2 (TPUs) supply the arithmetic-intensity / compute-vs-memory-bound framing everything else builds on; later chapters cover inference/serving (prefill vs decode latency) and training parallelism.

### 11. TurboQuant (research.google/blog; arXiv:2504.19874; ICLR 2026)
**Lead facts**: a training-free, model-agnostic **rotation-based vector quantization** method for KV-cache compression (and vector search). Two stages: **PolarQuant** (main compression) + **QJL** (1-bit error corrector).
- **Mechanism**: (1) apply a **random rotation** (QR of a Gaussian matrix) so each coordinate becomes ~N(0, 1/d) by concentration of measure; (2) **Lloyd-Max scalar quantization** per coordinate using a precomputed codebook of optimal Gaussian centroids (e.g. 3-bit → 8 centroids); (3) add **1 bit via Quantized Johnson-Lindenstrauss (QJL)** to unbias the residual and preserve inner products (attention scores). It is called "PolarQuant" because it converts coordinates to polar (radius/angle) form, eliminating the per-block normalization constants that add 1–2 overhead bits in standard vector quantization.
- **Results**: quality-neutral at **3.5 bits/channel**, marginal degradation at 2.5; TQ3 (3-bit) MSE 0.034, ~4.9× vs FP16; TQ4 3.8×; ~6× memory reduction overall with near-zero accuracy loss, no calibration or fine-tuning, on any transformer (validated on Llama-3.1-8B, Gemma, Mistral across LongBench, Needle-in-a-Haystack, ZeroSCROLLS, RULER, L-Eval).
- **vs NVFP4 / INT8 / INT4**: those are **fixed-grid scalar formats** applied per-tensor/block to weights/activations (and require storing full-precision block scales). TurboQuant is a **data-oblivious rotation + optimal scalar codebook + JL error correction** aimed specifically at the **KV cache** (and vector DBs), pushing below 3 bits where INT4/NVFP4 stop, and it is training-free — whereas NVFP4 in Nemotron is a *training* format, not a KV-cache compressor.

### 12. On-Policy Self-Distillation family
- **Thinking Machines "On-Policy Distillation" blog** (thinkingmachines.ai): the student generates its own rollouts (on-policy); the teacher does a single forward pass over them; the **per-token reverse KL** (log π_teacher − log π_student, evaluated on the same student-visited prefixes) becomes a dense per-token reward/advantage fed into the RL loss (discount factor 0, so each token optimizes only itself). Reverse KL is **mode-seeking** and "unhackable" (low KL ⇒ high teacher probability), reduces exposure bias, and — because you don't need a completed rollout to compute reward — lets you use short/partial rollouts. It is implemented on top of the RL script in Tinker, reusing sampling + the importance-sampling loss, with advantage = negative reverse KL. It penalizes the "forking tokens" that lead reasoning astray; the *final answer, though wrong, isn't penalized* because it's entirely predictable given the preceding sequence. On compute: per the Qwen3 Technical Report (arXiv:2505.09388, Table 21), on-policy distillation "achieves significantly better performance than reinforcement learning while requiring approximately only 1/10 of the GPU hours" (74.4% AIME'24); Thinking Machines Lab's own replication reports an even larger saving — "on the order of 50–100×," reaching teacher level in ~7–10× fewer gradient steps.
- **OPD lineage / true precursor**: **MiniLLM (Gu et al., arXiv:2306.13649)** first formalized OPD under a reverse-KL policy-gradient objective (mode-seeking argument); **GKD (Agarwal et al.)** unified on/off-policy data across divergences; Yang et al. (2026) later cast OPD as a special case of dense KL-constrained RL (teacher per-token log-ratio = implicit reward, and scaling that reward can push the student past the teacher).
- **arXiv 2411.14251 — CORRECTION (flag this in the interview)**: this ID is **"Natural Language Reinforcement Learning"** (Feng, Liu, Song et al., Nov 2024), which recasts RL constructs (value function, Bellman equation, policy iteration) into natural-language form via a **Language Value Function.** It is **NOT** a distillation paper and is **not** in the OPD citation chain. The reading-list assumption is incorrect; the intended precursor is almost certainly MiniLLM (2306.13649).
- The newer 2026 items (OPSD 2602.02482, SDPO 2601.20802, RLTF 2602.13949, 2601.18734) sit in the fast-moving 2026 OPD literature — themes include truncated reverse-KL with teacher top-K support matching, length-inflation fixes (Stable-OPD), and calibration fixes (CaOPD, which replaces confidence targets with student-grounded empirical success rates). These specific IDs were not individually verified against primary PDFs and should be double-checked against the awesome-on-policy-distillation index, since many 2026 papers share near-identical titles.

### 13. Broader context (1–2 innovations each)
- **DeepSeek-V3** (arXiv:2412.19437): MLA (compressed latent KV), fine-grained MoE with **auxiliary-loss-free load balancing**, MTP; FP8 training.
- **DeepSeek-R1** (arXiv:2501.12948): **origin of GRPO at scale**; R1-Zero shows pure-RL reasoning emerges without SFT; multi-stage recipe (cold-start SFT → reasoning RL → rejection-sampling SFT → general RL).

GLM-5 adopts DSA from here.
- **GLM-4.5** (predecessor): unified **ARC** (Agentic/Reasoning/Coding) MoE, 355B/32B active, established the slime + Muon recipe GLM-5 extends.
- **MiniMax-M1 / M2**: hybrid lightning-attention MoE, 1M context; **origin of CISPO**; efficient RL scaling ($534,700 full RL run). M2 continues the agentic/coding focus.

## Recommendations
1. **Memorize the clipping-comparison table** — it is the single highest-leverage artifact. If asked "how would you stabilize MoE RL," answer GSPO (sequence-level ratio) or IcePop (mask on train/infer mismatch); if asked "how do I keep exploration on rare reasoning tokens," answer CISPO (clip the IS weight, not the gradient); if asked "why does response length blow up," answer Dr.GRPO's length-normalization bias.
2. **Lead every model answer with the training-inference mismatch theme.** IcePop, CISPO, GSPO, K2.5's sign-agnostic clip, GLM-5's TITO/double-sided IS, and Nemotron's masked IS are all answers to "the rollout policy ≠ the training policy." Framing them this way shows synthesis, not memorization.
3. **For the optimizer question**, contrast concretely: AdamW (baseline) → Muon (orthogonalized updates, token-efficient) → GLM-5's **Muon Split** (per-head orthogonalization; no attention-logit clipping needed) vs Kimi's **MuonClip = Muon + QK-Clip** (clamps exploding attention logits). Do **not** assert "GLM resets optimizer states" — the sources do not support it; GLM's documented optimizer innovation is Muon Split, and I'd correct the premise if asked.
4. **For the 5-hour research project**, be ready to implement a GRPO variant from scratch and reason live about: advantage normalization (Dr.GRPO: drop length + std), clip design (DAPO Clip-Higher vs CISPO detached-weight vs GSPO sequence-level), token-vs-sequence loss aggregation, and async rollout (slime-style decoupling + partial rollouts). Rehearse the FLOPs/KV-cache formulas from Jax Scaling Book Ch. 4 for back-of-envelope model sizing.
5. **Benchmarks that change the recommendation**: if the target is a *dense* model, GSPO's advantage largely disappears — prefer DAPO/Dr.GRPO. If rollouts are cheap and near-on-policy, the mismatch fixes matter less. If the task is agentic/long-horizon, prioritize async infra (slime) and multi-turn credit assignment (CWM's masked-environment-token GRPO, K2.5's orchestrator-only PARL).
